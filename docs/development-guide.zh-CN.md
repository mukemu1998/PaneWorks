# PaneWorks 开发规范

本文档用于约定 PaneWorks 后续开发的代码组织方式。新功能开发、问题修复和重构都应遵守这里的边界，避免功能继续堆回少数大文件。

## 开发阶段

PaneWorks 后续按四个阶段推进：

| 阶段 | 目标 | 执行方式 |
| --- | --- | --- |
| 探索期 | 验证体验是否成立 | 可以快速试做，但要控制范围，并在功能稳定后整理结构。 |
| 稳定期 | 把已验证功能整理成清晰模块 | 先本地测试，再重构命名、文件边界和文档。 |
| 扩展期 | 按既有架构添加功能 | 新功能必须先判断归属层级，不允许随手塞进主窗口或主 ViewModel。 |
| 发布期 | 面向用户整理版本 | 本地验证通过后，再打包、写发布说明和上传包。 |

当前项目已进入稳定期与扩展期之间，后续默认按“先定落点，再写代码”的方式开发。

## 分层边界

### PaneWorks.App

负责 WPF 界面、用户输入、命令绑定、弹窗、托盘菜单和窗口事件转发。

允许放在这里：

- `MainWindow` 的 UI 事件桥接、窗口状态和覆盖层显示。
- `MainViewModel` 的界面状态、命令入口和用户提示。
- Dialog、Control、托盘菜单、主菜单和关于/设置窗口。

不应放在这里：

- 复杂窗口匹配算法。
- 复杂布局计算。
- 文件持久化细节。
- Win32 调用细节。
- 可复用的工作区恢复流程核心逻辑。

### PaneWorks.Core

负责稳定的数据模型、纯计算逻辑和业务规则。

适合放在这里：

- 布局节点、工作区绑定、显示器布局等模型。
- 不依赖 WPF、不依赖 Win32、不读写文件的纯算法。
- 未来可测试的服务接口或规则对象。

### PaneWorks.Infrastructure

负责系统能力和外部依赖。

适合放在这里：

- Win32 窗口枚举、移动、边框计算。
- JSON 持久化。
- Explorer 文件夹路径识别。
- 注册表、开机自启、进程启动等系统集成。

## 文件职责

每个文件应只表达一个明确职责。推荐命名方式：

| 前缀 | 用途 |
| --- | --- |
| `SnapAssist*` | 吸附预览、吸附落位、吸附兜底检测。 |
| `RuntimeLinkedResize*` | 运行时边缘联动、临时拓扑、磁吸和边界约束。 |
| `WorkspaceBinding*` | 工作区绑定、绑定显示、路径补全、绑定保存。 |
| `WorkspaceRestore*` | 工作区恢复、已启动窗口接管、窗口稳定化。 |
| `Workbench*` | 主菜单、最小化浮窗、菜单按钮动作。 |
| `Layout*` | 分区布局保存、编辑、删除、列表和未保存变更。 |

如果一个文件开始同时处理两个以上职责，应优先拆分。

## 行数红线

这些不是绝对规则，但后续开发应主动遵守：

- 200 行以内：健康。
- 250 行以上：需要检查是否能拆。
- 350 行以上：必须说明原因，通常应拆分。
- 单个方法超过 80 行：优先抽出子流程。
- UI、文件、Win32、算法混在同一个文件：必须拆分。

例外：`MainViewModel.State.cs` 这类纯绑定属性集中区可以略长，但不能加入复杂业务流程。

## 新功能流程

新增功能前先确认：

1. 这个功能属于 UI、Core 还是 Infrastructure。
2. 是否需要新增模型或保存字段。
3. 是否会影响现有布局、工作区或个人设置兼容。
4. 是否需要本地测试包验证。
5. 是否需要更新 README、CHANGELOG 或发布说明。

默认执行顺序：

1. 本地实现。
2. 本地构建验证。
3. 覆盖固定测试包。
4. 用户确认效果。
5. 用户明确要求“打包”后，再整理正式发布包和发布说明。

## 版本策略

- 文档、内部重构和开发规范调整不单独 bump 版本。
- 用户可见功能变化、兼容性变化、发布包变化才进入版本发布说明。
- 当前开发线保持 `v0.2.4`，正式发布前统一整理更新记录。
- 后续如果是小修小补，使用补丁版本；如果引入明显新能力，再提升次版本。

## 禁止事项

- 不把新功能直接塞回 `MainWindow.xaml.cs` 或 `MainViewModel.cs`。
- 不把临时调试布局、用户个人配置、`artifacts/` 打包产物提交进源码。
- 不为了行数好看硬拆破坏可读性。
- 不在未本地测试的情况下整理正式发布包。
- 不在用户未说“打包”前生成正式 release 上传包。

## 当前约定

从 `v0.2.4` 开发线开始，PaneWorks 后续开发严格按本文档执行。若未来发现规范不适合实际功能，应先修改本文档，再按新规范调整代码。
